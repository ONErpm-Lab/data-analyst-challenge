### Dado parcial de dezembro/2024

Os dados de dezembro/2024 estão incompletos. Apenas o YouTube reportou dados neste mês. O resultado é uma receita de US$ 2,05M em dezembro, contra uma média de US$ 6,3M nos outros meses de 2024. O volume de streams também confirma: 8,07 bilhões em dezembro contra uma média de 71–78 bilhões nos meses de outubro e novembro. Os números anuais de 2024 estão subestimados por este motivo, e a queda de -1,4% em relação a 2023 seria menor com o mês completo.

| Ano  | Receita Líquida | Crescimento YoY |
|------|----------------|-----------------|
| 2022 | US$ 57,87M     | —               |
| 2023 | US$ 72,17M     | +24,7%          |
| 2024 | US$ 71,17M     | -1,4%           |

**Projeção para 2025: US$ 79,46M (+11,7%)**

A projeção aplica a média dos dois crescimentos históricos disponíveis (+24,7% em 2023 e -1,4% em 2024), resultando em +11,7%. Aplicado sobre a receita de 2024: US$ 71,17M × 1,117 = US$ 79,46M.

Esse número deve ser interpretado com cautela por dois motivos: (1) apenas dois pontos de crescimento disponíveis tornam a média sensível a variações; (2) 2024 fechou em queda. Se o padrão de 2024 continuar em 2025, o crescimento real seria próximo de zero. O cenário de US$ 79,46M depende de recuperação nos principais mercados, especialmente América do Sul.

### 2. Melhor estratégia para potencializar receita

**Spotify — defender e ampliar**
Responsável por 54% da receita total de 2024 (US$ 38,5M). Apresentou leve queda sobre 2023 (US$ 39,7M → US$ 38,5M, -3%). É a plataforma com maior retorno por stream. Qualquer redução no catálogo ativo ou nos royalties do Spotify tem impacto imediato e proporcional na receita total.

**YouTube — principal alavanca de crescimento**
Cresceu de US$ 18,9M (2022) para US$ 23,8M (2024), um aumento de 26% em dois anos, e é a única grande plataforma com trajetória de crescimento ininterrupto. YouTube Red/Premium responde por parte relevante desse crescimento. Aumentar a presença e a qualidade do catálogo no YouTube é a iniciativa com melhor potencial de retorno no médio prazo.

**TikTok — oportunidade em crescimento**
Saiu de US$ 0,68M em 2022 para US$ 2,06M em 2024, triplicando de receita em dois anos. Ainda representa menos de 3% do total, mas a curva de crescimento é clara. Gêneros como Hip Hop/Rap e Trap, que têm boa conversão, têm alta penetração no TikTok.

**Facebook — volume sem retorno proporcional**
Facebook gerou 317 bilhões de streams em 2024, mas apenas US$ 4,1M em receita — a menor taxa de monetização entre todas as plataformas. Em 2022 eram US$ 6,3M, queda de 35% em receita apesar do crescimento de volume. O modelo de monetização do Facebook UGC não favorece distribuidoras. Não há indicativo de que investir mais em Facebook resulte em aumento de receita.

**Amazon — estável, sem destaque**
Cresceu de US$ 1,96M para US$ 2,68M em dois anos (+37%), mas representa menos de 4% da receita total e não apresenta potencial de volume relevante no curto prazo.

**Em resumo:** concentrar esforços em Spotify (defesa do volume de catálogo) e YouTube (crescimento ativo), com atenção ao TikTok como plataforma emergente. Facebook não merece investimento prioritário dado o desequilíbrio entre volume e receita.

## Critérios Técnicos

### Cálculo de crescimento YoY

- Para visão mensal: cada mês é comparado ao mesmo mês do ano anterior (ex: maio/2024 vs. maio/2023).
- Para visão anual: receita do ano atual vs. receita do ano anterior.
- Fórmula: `(receita_atual - receita_anterior) / receita_anterior × 100`
- Reversals (estornos com quantidade negativa) são identificados pela flag `is_reversal` e deduzidos para chegar na receita líquida (`net_revenue`). A receita bruta (`gross_revenue`) exclui reversals; a receita líquida os inclui.

### Projeção 2025

- Projeção global: média dos crescimentos YoY anuais disponíveis (2022→2023: +24,7% e 2023→2024: -1,4%) = +11,65%. Aplicada sobre a receita de 2024.
- Projeção por país: mesma lógica, calculada individualmente por `country_code` no mart `mart__revenue_region_goals`. Permite agregar para região e sub-região por soma no Power BI sem necessidade de cálculo adicional.
- A projeção está populada apenas para o ano mais recente (2024). Anos anteriores exibem nulo.

### Conversão streams × receita

- Métrica: `net_revenue / stream_quantity × 1.000` (receita líquida por 1.000 streams).
- Apenas eventos com `is_stream = true` são contados no denominador. Downloads são excluídos.
- Disponível por gênero no mart `mart__revenue_by_genre`.

### Agregação regional no dashboard

- `year_over_year_growth_pct` é um percentual. Ao agregar múltiplos países em uma região, a agregação correta é a **mediana**, não a soma nem a média. A mediana evita que outliers (mercados emergentes pequenos com crescimento de centenas de porcento) distorçam o número regional.
- Receitas, volumes de streams e projeções são agregados por **soma**.

## Como o Projeto Foi Realizado

### Stack

| Ferramenta | Função                                      |
|-----------|----------------------------------------------|
| DuckDB    | Banco de dados OLAP local                    |
| dbt       | Transformações e modelagem (staging → marts) |
| Python    | Export dos marts para Parquet                |
| Power BI  | Dashboard e visualizações                    |

**Por que DuckDB:** banco OLAP embutido, sem servidor, com suporte nativo a SQL analítico (window functions, FILTER, COPY TO PARQUET) e gratuito. Adequado para o volume de dados processado localmente.

**BigQuery foi considerado e descartado:** o volume dos arquivos inviabilizou o uso no ambiente local. A alternativa foi manter todo o processamento via DuckDB.

### Pipeline dbt

O pipeline segue três camadas:

**Staging (7 modelos)**

Leitura direta das 7 fontes CSV (2 de 2022, 2 de 2023, 3 de 2024). Os CSVs vieram sem nomes de coluna legíveis, com colunas nomeadas como `column0`, `column1` etc. A camada de staging renomeia para nomes de negócio (`store`, `date_of_event`, `quantity`, `revenue`, `country_code` etc.) e faz cast de tipos. Nenhuma regra de negócio é aplicada aqui.

**Intermediate (20 modelos)**

Dividida em três sub-camadas:

- *Cleaned*: deduplicação de eventos duplicados, criação da flag `is_reversal` (eventos com `quantity` negativa), e preenchimento de `country_code` nulo com `'NOT INFORMED'`. Segmentada em períodos para evitar erros de memória — ver nota abaixo.
- *Grouped*: agrega os eventos limpos por mês, somando `quantity` e `revenue` por `store`, `genre`, `country_code` e flags booleanas. Um modelo por ano (2022, 2023, 2024). A coluna `product` é descartada nesta camada pois não é necessária para nenhuma análise.
- *Final*: `int__events_final` é um UNION ALL dos três modelos grouped. É a única fonte usada pelos marts.

**Marts (6 modelos)**

| Mart                          | Grain                        | Finalidade                                       |
|-------------------------------|------------------------------|--------------------------------------------------|
| `mart__revenue_overview`      | mês × currency               | Visão executiva global de receita e streams      |
| `mart__revenue_by_store`      | store × mês × currency       | Receita e YoY por plataforma                     |
| `mart__revenue_by_region`     | country × mês × currency     | Receita e YoY por país, com hierarquia geográfica|
| `mart__revenue_by_genre`      | genre × mês × currency       | Receita, streams e conversão por gênero          |
| `mart__revenue_year_over_year`| ano × currency               | YoY anual e projeção global de 2025              |
| `mart__revenue_region_goals`  | country × ano × currency     | YoY anual e projeção de 2025 por país            |

Todos os marts incluem: `gross_revenue`, `net_revenue`, `stream_quantity`, `download_quantity`, `reversal_quantity`, `year_over_year_growth_pct` e `prior_year_net_revenue`.

**Testes dbt:** 338 definições em YAML, cobrindo `not_null` em todas as colunas obrigatórias e `accepted_values` nos booleanos (`is_stream`, `is_download`, `is_reversal`).

### Segmentação do intermediate por período

O processamento local gerou erros de falta de memória ao tentar processar os dados de 2022 e 2023 inteiros de uma vez. A solução adotada foi:

- 2022: 4 modelos, um por trimestre (Q1 a Q4)
- 2023: 4 modelos, um por trimestre (Q1 a Q4)
- 2024: Q1 e Q2 por trimestre; H2 (julho a dezembro) por mês individual, pois o volume de 2024 era maior ainda

Se o projeto fosse refeito, a segmentação seria por mês para todos os anos desde o início. Isso eliminaria a inconsistência de granularidade entre os períodos e tornaria o pipeline mais uniforme.

### Export para Parquet e conexão com Power BI

O script `scripts/export_marts_to_parquet.py` conecta ao banco DuckDB local e exporta cada mart como um arquivo `.parquet` em `data/exports/`. O Power BI lê esses arquivos diretamente via conector de pasta.

### Ambiente local vs. produção

Este projeto foi executado inteiramente em ambiente local. Em produção, as decisões seriam diferentes:

- **Infraestrutura:** os dados provavelmente já estariam em um cloud warehouse (BigQuery, Snowflake, Redshift). O DuckDB foi a alternativa viável para uso local.
- **Sem necessidade de segmentação manual:** com um warehouse em nuvem, não haveria restrição de memória. Todos os dados seriam processados de uma vez, sem quebrar em trimestres ou meses.
- **Pipeline automatizado:** o dbt seria executado por um orquestrador (Airflow, dbt Cloud, Prefect) com agendamento definido. Aqui foi executado manualmente.
- **Conexão direta ao warehouse:** o Power BI se conectaria ao warehouse via conector nativo, sem necessidade de exportar Parquet manualmente.
- **Dados já estruturados na origem:** em produção, os dados provavelmente chegariam com schema definido e nomes de coluna legíveis, sem os problemas dos CSVs sem cabeçalho que encontramos.
- **Qualidade de dados na ingestão:** deduplicação, tratamento de nulos e validação de tipos seriam tratados no pipeline de ingestão, não na camada intermediate do dbt.

## Como Executar o Projeto

### Requisitos

- Python >= 3.13
- [uv](https://docs.astral.sh/uv/) (recomendado) ou pip
- Power BI Desktop

### 1. Instalar dependências

```bash
uv sync
```

Ou com pip:

```bash
pip install dbt-core>=1.11.11 dbt-duckdb>=1.10.1 duckdb>=1.5.4
```

### 2. Configurar o perfil do dbt

Adicione o bloco abaixo ao arquivo `~/.dbt/profiles.yml`, ajustando o caminho absoluto para o banco de dados:

```yaml
analytics_dbt:
  target: dev
  outputs:
    dev:
      type: duckdb
      path: /caminho/absoluto/para/data/database/onerpm_analytics.duckdb
      threads: 4
      schema: main
      settings:
        preserve_insertion_order: false
```

### 3. Baixar os arquivos de dados

Faça o download dos arquivos `.gz` listados na seção **Dados** do README principal e coloque-os em `data/raw/`.

### 4. Carregar os dados brutos no DuckDB

```bash
python scripts/load_files_to_duckdb.py
```

O script lê todos os `.gz` em `data/raw/` e cria uma tabela por arquivo no banco DuckDB. DuckDB lê arquivos comprimidos diretamente, sem necessidade de descompactar.

### 5. Executar o pipeline dbt

```bash
cd analytics_dbt
dbt seed        # carrega o seed de países (countries.csv)
dbt run         # constrói todos os modelos (staging → intermediate → marts)
dbt test        # valida os 338 testes definidos nos YMLs
```

### 6. Exportar os marts para Parquet

```bash
cd ..
python scripts/export_marts_to_parquet.py
```

Os arquivos serão gerados em `data/exports/`, um `.parquet` por mart.

### 7. Conectar no Power BI

Abra o Power BI Desktop e aponte a fonte de dados para a pasta `data/exports/`. Todos os marts estarão disponíveis como tabelas separadas.
